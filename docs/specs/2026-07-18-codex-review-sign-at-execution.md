## 1. Findings

### P0 blocker

1. **The proposed amendment guard permits signature smuggling that the current UI will badge as human-signed.** Section 7 rejects only a signature/amendment trailer at the end of the normalized body ([spec:56](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:56)). A valid-looking amendment can therefore contain `Signed off: BK, 2026-07-18` inside the body, followed by ordinary content, and finish with the amendment trailer. It passes the proposed “no stacking” test because the normalized body ends in ordinary content. The TS parser finds `Signed off:` anywhere via a multiline regex ([parse.ts:335](/Users/bk/github/research-plans/board/src/lib/parse.ts:335)); the action state repeats that fallback ([actions.ts:51](/Users/bk/github/research-plans/board/src/lib/actions.ts:51)); PlanReader then renders the green badge ([PlanReader.tsx:641](/Users/bk/github/research-plans/board/src/views/PlanReader.tsx:641)), and Tracker reports “Signed off” ([Tracker.tsx:500](/Users/bk/github/research-plans/board/src/views/Tracker.tsx:500)). Shared payloads include every canonical `v*.md` ([board.py:681](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:681)), so a collaborator can concretely be shown a forged signed badge. An `/execute` resolver that reuses this parsing could also skip the gate [inference].

   **Suggested spec amendment:** Require one shared strict EOF trailer parser across the hook, TS, badges, and `/execute`; reject any signature/amendment marker outside the single canonical final trailer and add an interior-signature adversarial test.

2. **The amended-vN → signed-vN+1 recommitment is not the same-content transaction claimed by the spec.** Tickets are bound to the exact target slug and version ([signoff_gate.py:73](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:73)) as well as the normalized content hash ([signoff_gate.py:96](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:96)). The filename seam itself is correct: `board.py` writes `.import-approved-<slug>-v<N>` under `plans/execution` ([board.py:2359](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:2359)), and the hook derives that same path ([signoff_gate.py:309](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:309)). A ticket for N+1 is therefore possible. The problem is the candidate: the spec says to serve amended vN content and write it as signed vN+1 ([spec:41](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:41)), but the body carries `Execution Plan v<N>` and `Supersedes: v<N-1>` metadata ([execution-plan.md:1](/Users/bk/github/research-plans/skills/managing-research-plans/templates/execution-plan.md:1)). Copying it verbatim makes a malformed vN+1; correcting those fields changes the normalized hash. The existing scorecard migration is also only `.draft-vN` → `vN`, not vN → vN+1 ([review.md:15](/Users/bk/github/research-plans/commands/review.md:15)). Current batch approval is source/target-coupled to the newest draft’s same version ([board.py:1397](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1397)).

   **Suggested spec amendment:** Materialize and display a real `.draft-v<N+1>.md` candidate with updated title and `Supersedes`, score that exact candidate, then mint the N+1 ticket over it; the hash function can be reused, but not the raw amended-vN hash.

### P1 material

3. **The ticketless amendment path cannot establish its claimed “after execution” provenance.** The stated invariant says the trailer appears iff the system auto-finalized a post-execution revision ([spec:21](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:21)). The hook receives only tool name, tool input, file path, and cwd ([signoff_gate.py:160](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:160)); its enforcement point knows the target version and whether prior files exist, but no execution identity, result bundle, tracker state, or trusted `/sync` capability ([signoff_gate.py:263](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:263)). An agent can therefore create amendment v2 immediately after signed v1, before any execution, and satisfy every proposed condition [inference]. The badge would not claim human approval, but the “recorded after execution” assertion would still be false and shareable.

   **Suggested spec amendment:** Gate amendment writes with a trusted short-lived capability minted by the execution/sync workflow, or rename the trailer to an agent-recorded claim that does not assert execution timing.

4. **Sign mode conflicts with an already-open persistent board because every mode uses the same lock.** `serve()` always acquires `plans/.board.lock` before examining gate/batch mode ([board.py:1071](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1071)); any live owner causes a hard refusal ([board.py:931](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:931)). The dashboard deliberately has no idle timeout ([board.py:1505](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1505)), while `/board` presents it as persistent ([board.md:15](/Users/bk/github/research-plans/commands/board.md:15)). The existing hook already warns that an open board prevents its modal gate ([signoff_gate.py:418](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:418)). `--force` merely overwrites the lock and is not a safe handoff.

   **Suggested spec amendment:** Define controlled dashboard shutdown/handoff or separate sign-session locking/ports, including behavior when `/execute` starts while a board is open.

5. **`/sign` has no coherent source-resolution contract.** The spec says amendment timeouts can be recovered with `/sign` ([spec:41](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:41)), but defines `/sign` as scanning pending drafts only ([spec:44](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:44)). The reusable batch helper likewise scans only newest `.draft-vN` files ([board.py:2424](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:2424)). A repository-wide scan would also include archived-only components, which the current UI explicitly excludes from action eligibility ([actions.ts:31](/Users/bk/github/research-plans/board/src/lib/actions.ts:31)); `/renew` promises those remain browsable but never nagged about ([renew.md:15](/Users/bk/github/research-plans/commands/renew.md:15)). Tickets carry no source kind/path/version mapping beyond target slug/version/hash ([board.py:2363](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:2363)).

   **Suggested spec amendment:** Define a single current-tracker eligibility resolver returning `sourceKind`, `sourcePath`, `sourceVersion`, and `targetVersion`; include amendments and valid-ticket recovery, and exclude pre-renewal-only groups.

6. **The new workflow omits the finalization transaction, causing tracker and no-argument `/execute` regressions.** Current approval routing copies the exact draft, appends the trailer, removes the ephemeral draft, migrates/reviews the scorecard, updates the plan link, preserves revision status, logs the decision, and offers execution ([board.md:31](/Users/bk/github/research-plans/commands/board.md:31)). The proposed `/sign` mentions only writing versions and scoring migration ([spec:44](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:44)). More immediately, `/plan` will leave a draft-ready row `not started`, while no-argument `/execute` currently proposes the first `planned` row and treats “no signed plan” as an error ([execute.md:9](/Users/bk/github/research-plans/commands/execute.md:9)). It can therefore skip the draft the researcher just authored. Adopted rows also cannot be blindly reset to `planned`.

   **Suggested spec amendment:** Specify an atomic post-ticket transaction and revise no-argument `/execute` to consider current-tracker pending drafts/amendments while preserving provenance-specific tracker status.

7. **The proposed timeout/Ctrl-C and Request-changes contract is not supplied by existing batch machinery.** The spec says Request changes writes a feedback document and that completion, timeout, or Ctrl-C inherit the batch exit-0 contract ([spec:32](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:32)). Current batch rejection is only appended to process memory ([board.py:1463](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1463)); the UI explicitly says rejections do not survive a crash ([BatchGate.tsx:336](/Users/bk/github/research-plans/board/src/views/BatchGate.tsx:336)). Timeout exits 0 only for batch mode ([board.py:1518](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1518)); Ctrl-C exits 130 ([board.py:1534](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1534)). The batch UI has a single textarea rather than target-scoped annotations ([BatchGate.tsx:288](/Users/bk/github/research-plans/board/src/views/BatchGate.tsx:288)), whereas normal approval blocks while target-draft comments remain unsent ([actions.ts:35](/Users/bk/github/research-plans/board/src/lib/actions.ts:35)).

   **Suggested spec amendment:** Define durable per-draft feedback/annotation storage, unresolved-comment blocking, caller-visible rejection recovery, and exact timeout/Ctrl-C exit semantics instead of inheriting the current batch contract.

8. **Removing trailer-in-draft validation contradicts the no-migration promise.** The spec identifies the current template placeholder as the source of trailer-bearing drafts ([spec:67](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:67)), then says existing pending drafts need no migration ([spec:75](/Users/bk/github/research-plans/docs/specs/2026-07-18-sign-at-execution-design.md:75)) while deleting both trailer guards. Current routes deliberately reject such drafts because normalization cannot distinguish trailer identity ([board.py:1309](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1309), [board.py:1435](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/board.py:1435)). Existing or manually authored drafts can therefore reach sign mode with placeholders or forged trailers, producing hash mismatches or stacking.

   **Suggested spec amendment:** Keep strict trailer validation in sign mode and add an explicit repair/error path for legacy pending drafts; removing the template placeholder is prevention, not migration.

9. **Results capture still binds bundles to “latest signed” rather than the latest governing amendment.** `/sync` will auto-finalize an amendment and then continue to results capture, but `results.md` stamps `manifest.planVersion` from the latest signed version ([results.md:19](/Users/bk/github/research-plans/commands/results.md:19)) and runs conformance only when a signed plan exists ([results.md:21](/Users/bk/github/research-plans/commands/results.md:21)). The validator template likewise requires a signed plan ([rp-results-validator.md:3](/Users/bk/github/research-plans/skills/managing-research-plans/templates/agents/rp-results-validator.md:3)). The deviation loop currently rebinds the staged bundle to the newly finalized revision ([execution-loop.md:40](/Users/bk/github/research-plans/skills/managing-research-plans/references/execution-loop.md:40)); an amendment-only canonical version makes that wording and validation eligibility ambiguous. A manual `/results` after `/sync` could bind the bundle to the previous human-signed version.

   **Suggested spec amendment:** Define “governing plan version” as the latest canonical signed-or-amendment version throughout results capture, validation, and result-validator prompts; keep human approval as separate metadata.

10. **The absolute “nothing executes ungated” and “Signed off iff human click” invariants exceed the hook’s documented enforcement boundary.** The hook matches only Write/Edit ([signoff_gate.py:176](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:176)) and explicitly permits `RESEARCH_PLANS_NO_GATE=1` ([signoff_gate.py:273](/Users/bk/github/research-plans/skills/managing-research-plans/scripts/signoff_gate.py:273)). Public documentation also acknowledges Bash-mediated writes are outside the matcher and prescribes that bypass for headless/CI ([reference.md:96](/Users/bk/github/research-plans/docs/reference.md:96)); the skill repeats it ([SKILL.md:46](/Users/bk/github/research-plans/skills/managing-research-plans/SKILL.md:46)). The proposed invariants are therefore false for supported execution modes.

   **Suggested spec amendment:** Scope the invariants to non-bypass Write/Edit workflows and specify the unsigned/amendment behavior for headless/CI, or replace the bypass with a non-signing capability path.

## 3. Risks and tradeoffs for BK

- Lazy signing weakens the time-separated commitment signal: the human approves at the moment of consequence rather than after a separate review interval. The execute prompt immediately commits afterward ([execution-loop.md:9](/Users/bk/github/research-plans/skills/managing-research-plans/references/execution-loop.md:9)), which preserves the mechanical `uncommitted` criterion but changes its substantive meaning.

- Auto-amendments make canonical plan history partly agent-authored rather than human-approved. That can be honest if the badge and trailer remain unambiguous, but it raises the security importance of strict trailer parsing and collaborator-facing provenance.

- Re-signing an amendment as a new version creates two substantively similar canonical versions, extra scorecard migration, and result-version churn. Attaching later approval metadata to immutable amendment vN may be simpler than manufacturing signed vN+1.

- The persistent/shared board still has a clear role without Approve: remote and hosted boards already cannot expose live researcher actions because actions are live-only ([actions.ts:19](/Users/bk/github/research-plans/board/src/lib/actions.ts:19)); comments, review requests, reports, reopen, history, and collaborator reading remain useful. The new amendment badge becomes more—not less—important in those modes.

- `/report` is structurally compatible once bundle attribution is correct: it resolves the plan directly from `manifest.planVersion` without requiring a signature trailer ([report.md:9](/Users/bk/github/research-plans/commands/report.md:9)). The regression is upstream in results capture and validation, not report generation itself.

- Scorecard integrity remains ordering-dependent. Drafts intentionally omit `uncommitted` ([review.md:9](/Users/bk/github/research-plans/commands/review.md:9)), and migration skips rescoring ([review.md:15](/Users/bk/github/research-plans/commands/review.md:15)). That is correct only if the signed-at-execute transaction reliably commits before governed work; `/sign` by itself does not do so [inference].
