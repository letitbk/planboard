# Sign-at-execution: slim owned gate, lazy sign-off policy, post-execution amendments

Date: 2026-07-18
Status: rev 2 — codex review folded (docs/specs/2026-07-18-codex-review-sign-at-execution.md); ready for plan
Origin: the p2p trailer-in-draft failure (napkin Domain Notes 2026-07-18) prompted a comparison with plannotator's approval model. Four decisions were locked with BK via AskUserQuestion rounds; three further forks were decided by BK after the codex review (§12).

## 1. Problem

Approval in research-plans is a stateful, board-mediated transaction: draft on disk, persistent server with a frozen boot payload, content-hash staleness checks, trailer guards, forgery-protected tickets, exit-code routing, and relaunch choreography. Each guard is individually justified, but together they make the approval moment the plugin's largest edge-case surface (frozen-payload approve, trailer-in-draft 400, stale-draft 409, gate-timeout recovery routing, `--gate-batch`/`--allow-single`). plannotator demonstrates the opposite pole: approval as a near-stateless one-shot browser decision bound to the moment it matters. Separately, the click burden is a policy problem independent of mechanism: today every plan version wants a human sign-off at authoring time (v1 at `/plan`, every revision at `/sync`, every adopt batch), which BK finds too heavy.

## 2. Decisions locked (BK, 2026-07-18)

1. **Board role:** approval moves out of the board; the board keeps everything else (tracker, plan reading + diffs, results, reports, scorecards, timeline, sharing, and its other mutating routes: reopen, verdict, review requests, model profile).
2. **Mechanism:** a slim owned one-shot gate UI evolved from the existing gate machinery — not a plannotator dependency, not a native plan-mode hook.
3. **Policy:** sign at execution, not authoring. Drafts accumulate gate-free; the gate fires once per component at the point of consequence. `/sign` exists for optional early commitment.
4. **Post-execution revisions:** auto-recorded amendments with an honest non-approval trailer; no gate, no click. Re-execution gates the amended content into a new signed commitment.

Post-review forks (BK, 2026-07-18, after codex): re-commitment materializes a real signed v(N+1) via a draft candidate; the amendment trailer is `Amendment recorded, <date>` with no timing claim; sign sessions take the board lock via a token-authorized shutdown handoff. Details in §12.

## 3. Semantics — four rules

1. **`Signed off: <name>, <date>` appears iff a human clicked a gate.** The ticket/exit-code machinery keeps this mechanically unforgeable within the gated Write/Edit workflows. Scope honestly stated: the hook matches only Write/Edit (signoff_gate.py:176), and the documented headless/CI escape (`RESEARCH_PLANS_NO_GATE=1`, signoff_gate.py:273; reference.md:96; SKILL.md:46) plus Bash-mediated writes sit outside the matcher — that posture is unchanged by this design and the invariant claims nothing about it.
2. **`Amendment recorded, <date>` appears iff the system auto-finalized a revision without a gate.** It claims recording only — no human approval and no execution timing (the hook cannot verify "after execution"; timing evidence lives in git history and the decision log). sync.md's doctrine survives verbatim: a recorded revision is an amendment; a silent deviation is a breach. What changes is only that recording no longer demands a retroactive signature.
3. **One strict trailer grammar, everywhere, reject-not-ignore.** A plan document is well-formed iff its last non-empty line is exactly one canonical trailer (signature or amendment form) — or none, for drafts — and NO other line in the document matches either trailer pattern. One shared grammar definition, implemented on both sides per the payload/parse duplication rule (Python for the hook and payload; TS for parse.ts/actions.ts) and pinned by contract tests. The hook DENIES any plan write violating the grammar (including interior `Signed off:` lines in signed or amendment writes); the board renders a violating on-disk file with a `malformed trailer` warning badge — never `signed`. This closes the interior-signature smuggling hole (codex P0-1: parse.ts:335 and actions.ts:51 today detect `Signed off:` on any line via `/m` regex and would badge a forged interior signature green).
4. **Nothing executes ungated** — within the supported interactive workflows (same scope statement as rule 1). `/execute` targets the component's latest version; if that version is an unsigned draft or an amendment, the slim gate fires first and mints the signature. A component whose latest version is already signed runs with zero clicks.

Trailer taxonomy: `Signed off: <name>, <date>` (human clicked: /sign, the /execute gate, /adopt batch) and `Amendment recorded, <date>` (auto-finalized by /sync or the execution loop's deviation remedy). No third form.

## 4. The slim gate (sign mode)

A new one-shot mode of board.py, following the existing pattern where `gate` and `batch` are payload keys (board.py:1077): payload key `sign`, same built template, App renders a dedicated **SignOffView** via the existing early-return mode boundary (App.tsx:182 pattern).

**Page content per pending item:** rendered plan body, diff vs the last canonical version, the item's scorecard chips (drafts are scored at draft time per review.md), per-item annotations, and two actions: **Approve** and **Request changes**. Batch payloads list all pending items with per-item decisions (absorbs `--gate-batch`).

**Build honesty (codex P2-11):** this is more than a BatchGate rename. `PlanBody` is currently private to PlanReader (PlanReader.tsx:828) and must be extracted into a shared component; BatchGate renders raw Markdown with a single shared textarea (BatchGate.tsx:268, :288) and its rejections live only in process memory (BatchGate.tsx:336). SignOffView reuses PlanBody/DiffView/AnnotationLayer/ScorePanel and carries sign-specific annotation and recovery state as its own explicitly planned work.

**Protocol — tickets for everything.** Approve re-reads the source from disk, refuses on content-hash mismatch vs the served copy, then writes the same durable `.import-approved-<slug>-v<N>` ticket the batch route writes today (board.py:2359; hook derives the same path, signoff_gate.py:309). **Request changes persists per-item feedback to disk** (per-draft feedback documents with the item's annotations — never memory-only; this replaces the current batch behavior where rejections do not survive a crash). Approve is blocked while the item has unsent annotations (the actions.ts:35 semantic, now per item). Exit semantics, stated exactly: all-decided or timeout → exit 0 with a summary (tickets and persisted feedback are the durable record; callers MUST enumerate them on disk and never trust stdout alone — plan.md's existing batch rule becomes the universal rule); Ctrl-C → exit 130, same durable state. Recovery from any interruption is: relaunch via `/sign`, which re-offers undecided items and completes outstanding tickets (§5).

**Strict validation stays (codex P1-8).** Sign mode refuses to serve a source that violates the trailer grammar. For the known legacy case — a pending draft carrying the template's placeholder trailer — the workflow performs a visible one-line repair first (strip the placeholder from the mutable draft, tell the researcher), then serves it. Removing the template placeholder (§9) is prevention; this repair path is the migration.

**Lock handoff (codex P1-4).** All board.py modes share `plans/.board.lock` (board.py:1071, hard refusal at :931), and the persistent dashboard has no idle timeout. board.py therefore gains `POST /api/shutdown` in the existing local-mutation route class (127.0.0.1-bound, per-boot board_token required, local_request_ok — same hardening as the other local mutation routes): it triggers the clean `done.set()` exit. When a sign session (or the hook's interactive gate) finds a live lock owner, it reads the board bookkeeping file for port+token, POSTs shutdown, waits bounded for lock release, then proceeds; the researcher's old board tab lands in the existing "sleeping" reconnect state. A stale lock keeps its existing recovery. Hosted/remote boards are unaffected (no lock, no local routes).

**What sign mode does NOT have:** no persistent server life, no reload/second-tab reconciliation, no `/api/feedback` approve, no relaunch choreography. The hook gate remains the backstop for direct `v<N>.md` writes and keeps its exit-code protocol (exit 2 timeout, exit 0 authorize), now serving the SignOffView (single-item sign payload) instead of the full board.

## 5. Command flow changes

**The finalization transaction (defined once, cited everywhere — codex P1-6).** A single reference section (new `references/sign-off.md`, replacing the duplicated choreography paragraphs in plan.md/sync.md/board.md) defines the post-ticket transaction every caller runs per approved item: copy the exact approved draft bytes → append the signature trailer → write `v<N>.md` (hook validates the ticket) → delete the ephemeral `.draft-v<N>.md` (keep `v<N>-draft-K.md` snapshots) → run the review workflow (scorecard migration draft→signed, same version, review.md:15 semantics unchanged) → update the tracker plan link → tracker status per caller rules below (never regressing — the existing finalize-never-regresses rule) → decision-log entry. Cross-file references use the named section, not step numbers (the fragile-step-refs cleanup rides along for the touched files).

- **`/plan`.** Step 6 collapses: write the draft (no trailer), score it, set the tracker row `planned` with the plan link pointing at the draft — `planned` now means "has a plan (draft or signed)", which keeps no-argument `/execute` proposing the row the researcher just authored (codex P1-6's regression). Message: "draft ready — it signs at `/execute`, or run `/sign` to commit it now." The post-finalize chain becomes a post-draft chain offering `/execute`. Opening the board to read/annotate stays optional; it is review, not approval.
- **`/execute`.** "Latest version" = highest N across `v<N>.md` and `.draft-v<N>.md` (existing numeric-newest rule). Entry per component: latest is signed → proceed as today. Latest is a pending draft → sign session on it (with lock handoff if a board is open) → finalization transaction → the standard execute prompt runs unchanged (execution-loop.md; its part-1 commit consent preserves the `uncommitted` flag semantics). Latest is an amendment → **materialize a real `.draft-v<N+1>.md` re-commitment candidate** (codex P0-2): copy the amendment, update the title version and `Supersedes: v<N> — re-commitment for re-execution`, update the rp-model marker's reported side, score it, then sign-session it like any draft — the ticket is minted over the candidate itself; no hash reuse across versions (plan bodies embed their version in the title and Supersedes line, so rev 1's "zero new hash machinery" claim was wrong and rev 1's normalize_plan change is withdrawn). Request changes → revise → relaunch within the flow. Timeout/undecided → skip that component with "re-run /execute or /sign"; others proceed. No-argument resolution proposes the first `planned` row, which now includes draft-ready components.
- **`/sync`.** Material deviation → build the revision draft exactly as today (Supersedes line, provenance marker, drafting-trail snapshots), score it, then **auto-finalize** as `v<N+1>.md` with the amendment trailer via the hook's amendment path (§7). No board open, no click, tracker status untouched. Results capture downstream binds to the **governing plan version** (codex P1-9): the latest canonical version, signed or amendment — `results.md`'s `manifest.planVersion` stamping (:19), plan-conformance eligibility (:21), the rp-results-validator template's signed-plan requirement, and execution-loop.md:40's rebinding wording all adopt this term. Human approval state is separate metadata, never a validation-eligibility gate.
- **`/adopt`.** The batch review room becomes a sign session in batch mode; `--gate-batch`/`--allow-single` disappear. Approved plans get `Signed off:` trailers (a human genuinely clicked; the plan body's `Provenance: retrospective` carries the retro honesty). Tracker status of adopted rows is preserved, never reset to `planned`. Unapproved drafts stay drafts, signable later via `/sign` — no resumed-batch special case.
- **`/sign` (new command) — source-resolution contract (codex P1-5).** Resolver runs over CURRENT tracker components only (pre-renewal/archived groups are excluded, matching actions.ts:31; they stay browse-only permanently). It returns per component `{sourceKind: draft | amendment | ticket, sourcePath, sourceVersion, targetVersion}`. Default scope: pending drafts (newest `.draft-v<N>`), plus **outstanding-ticket recovery** — a valid unexpired ticket whose `v<N>.md` was never written gets its finalization transaction completed (this owns the interrupted-session recovery story). Amendments are offered for re-commitment only when the researcher names the component explicitly (`/sign <component>`) — an amendment-only component is a settled record, not a nag. `/sign` owns the full finalization transaction including the decision-log entry, and ends by suggesting `/execute` for newly signed planned components (a message, not an auto-chain).
- **`/board`.** Loses the approval role; gains nothing but the shutdown route and badge changes. Everything else unchanged.

## 6. Board changes (dashboard side)

Delete: the Approve/Request-changes actions and their `/api/feedback` decision routing (board.py ~1280-1330), the trailer-in-draft 400 guards (:1316, :1441 — superseded by the strict grammar, which is enforced at write time and validated at sign-serve time), the stale-approve exit-4 relaunch path, and the approve→signoff-order→relaunch choreography in board.md. The persistent board keeps its other mutating routes untouched (reopen, verdict, review requests, model profile) and keeps annotation → collect feedback loops.

Add: trailer-derived plan badges from the strict grammar — `signed ✓`, `amended △`, `malformed trailer ⚠` (never silently signed) — surfaced wherever signed-state shows today (PlanReader header :641, Tracker :500, Timeline); a pending draft shows "pending — signs at /execute or /sign". `actionsVisible` and its helper survive for the remaining live actions (review/reopen/model), only the approve branch dies.

**Retained deliberately (codex P2-12):** the hostile-ingress sanitizer keeps `signoff` among the action keys stripped/neutralized from hand-delivered collaborator documents (board.py:2063, :2152) with its tests (test_board.py:2723) — removing the live producer does not make legacy or hostile `SIGNOFF` input safe to route.

## 7. Enforcement changes (signoff_gate.py)

- **Strict trailer grammar (shared).** One grammar function: canonical final trailer (signature | amendment | none), zero interior occurrences of either pattern anywhere else in the document. The hook applies it to EVERY gated plan write (signed, amendment, and draft-path writes it inspects); Python payload collection and TS parse implement the same grammar; a contract test pins all implementations to shared fixtures (well-formed signed, well-formed amendment, interior-signature attack, stacked trailers, legacy placeholder draft).
- **normalize_plan: unchanged.** Rev 1's amendment-stripping extension is withdrawn — re-commitment now flows through a real draft candidate, so ticket hashing only ever compares a trailer-free draft against draft-plus-signature, which the existing normalize already handles (:38-49).
- **New allowed path — amendment writes.** A `Write` of `plans/execution/<slug>/v<N>.md` passes without a ticket iff ALL hold: (a) the file does not already exist (immutability of existing versions untouched); (b) `v<N-1>.md` exists (amendments are revisions, never a first version); (c) the document satisfies the strict grammar with final trailer = the amendment form. Forgery analysis: the only claim the trailer makes is "an ungated revision was recorded" — a claim the agent is the legitimate author of. It never claims human review or execution timing; the board badges it `amended △`; executing it still requires a human gate on a materialized candidate.
- **Signature writes:** unchanged (ticket validation with slug+version+hash binding per check_ticket :60-102, forgery guard on `.import-approved-*` :245, interactive backstop gate — now opening the SignOffView). Error and recovery messages throughout the script that currently route to "the board" are rewritten to route to `/sign` (signoff_gate.py:408 area).

## 8. Scoring and integrity flags

review.md's machinery fits with two edits: the trigger list "after a sign-off (by /plan, /sync, /adopt, or a board Approve)" becomes "after a sign-off (by /sign, the /execute gate, /adopt) or an amendment finalize (/sync)"; and target resolution treats a re-commitment candidate `.draft-v<N+1>.md` as an ordinary draft (scored like any draft; migration stays same-version draft→signed at finalize — no vN→vN+1 migration exists or is needed). Amendment versions are scored on finalize like signed ones. The `uncommitted` flag keeps working (the execute prompt commits the freshly signed plan before running); the spec notes honestly that `/sign` alone does not commit — the flag's guarantee attaches to the execute path, and a `/sign`-then-manual-execution flow can still incur the `uncommitted` flag. `unrecorded-deviation` gets sharper: with recording free of clicks, an unrecorded deviation is less excusable.

## 9. Template fix (rides along)

`templates/execution-plan.md` currently ends with a literal `---` + `Signed off: <researcher name>, <YYYY-MM-DD>` placeholder — the direct cause of the p2p trailer-in-draft failure. Remove it, and update the template's guidance comment (execution-plan.md:8) plus the trailer-related wording (:85) to the new taxonomy. The trailer is appended only by the finalization transaction. With drafts structurally trailer-free, the strict grammar simply rejects the legacy stragglers into the §4 repair path.

## 10. What this deletes (edge-case ledger)

Frozen-payload approve semantics and its 409 stale-draft dance; trailer-in-draft guards as standalone checks (subsumed by the grammar); `--gate-batch` + `--allow-single` and the resumed-batch special cases; gate-timeout recovery routing into the persistent board (recovery is "relaunch /sign"); the plan.md/sync.md/board.md approve choreography paragraphs (replaced by one named reference section); the in-board Approve surface and its rescoring trigger. The persistent board's remaining POST routes and their guards stay as-is.

Removing `--gate-batch` is a CLI-flag break with no known external consumers; project napkins that memorized old flows get post-release guidance (the v0.16 pattern).

## 12. Resolved design calls

- `/execute` runs the **latest** version, gating it if unsigned; it never silently falls back to an older signed version.
- Gate-then-execute-prompt ordering: the sign session runs first, then the standard one-question execute prompt unchanged. Folding execute consent into the gate's Approve is a possible future slimming, deliberately out of scope.
- Adopt keeps `Signed off:` (trailer records what actually happened — a click). Adopted rows keep their tracker status.
- Sign mode uses tickets for all planned flows; the hook backstop keeps its exit-code protocol. One UI, two transports, both pre-existing.
- **BK fork 1 (2026-07-18):** re-commitment of an amendment materializes a real `.draft-v<N+1>.md` candidate (updated title/Supersedes/marker), scored and ticketed like any draft — no approval-metadata sidecar on immutable vN, one mechanism for everything canonical.
- **BK fork 2:** the amendment trailer is `Amendment recorded, <date>` — recording claim only; no execution-timing assertion (the hook cannot verify one) and no trusted-capability machinery.
- **BK fork 3:** board-lock contention resolves via the token-authorized local `/api/shutdown` handoff; sign sessions close a live board cleanly and take the lock.
- Pre-renewal/archived components are permanently browse-only: excluded from `/sign` and from amendment re-commitment.
- Request-changes feedback is durable by contract: survives timeout, Ctrl-C, and browser close; `/sign` relaunch re-offers it.

## 13. Non-goals

No plannotator plugin dependency; no ExitPlanMode/PermissionRequest hook; no changes to reopen/verdict/review/model-profile board routes; no changes to sharing/hosted flows (remote boards never had live actions — actions.ts:19); no per-project sign-off policy setting; no approval-metadata sidecar files; no trusted post-execution-timing capability; no change to the documented headless/CI `NO_GATE` posture; no durable seed-queue redesign.

## 14. Risks accepted (surfaced to BK)

Lazy signing weakens the time-separated commitment signal — approval happens at the moment of consequence rather than after a separate review interval; mitigations are `/sign` for early commitment and draft-time scorecards keeping review quality visible early. Auto-amendments make part of canonical plan history agent-authored; the strict grammar and unambiguous badges are load-bearing for that honesty, which is why grammar violations reject rather than degrade. Re-commitment creates a near-duplicate canonical version (v(N+1) ≈ amendment vN plus signature); accepted as the price of one mechanism.
