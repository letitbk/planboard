- [ ] **Step 1: Write the file** (full content):

```markdown
# The execution loop

Referenced by `/research-plans:execute` and the post-finalize chain in `/research-plans:plan`. One human question starts execution; after that, capture, validation, report, and bookkeeping run without stopping — the only interruption is the deviation stop below. `/research-plans:sync` remains the manual recovery checkpoint for work done outside this loop.

## The execute prompt

Ask ONE AskUserQuestion (never re-ask pieces separately) with three parts:

1. **Execute now, or later?** A yes includes consent to COMMIT the signed plan (and its scorecard) first — say so in the question text: the rubric's `uncommitted` integrity flag stays meaningful only when the plan is committed before the work it governs. On "later", stop here; `/research-plans:execute <component>` re-enters this prompt any time.
2. **Which model?** Pre-select the profile's `execute` row (`models.py stage execute`); offer the standard alternatives (opus / sonnet / haiku / a custom id). When the choice differs from the model you are running as, print the standard nudge line (`Model profile: …  Switch with /model <model> …`) and WAIT for the researcher to switch before starting — they opted into that model, so waiting is correct here (the only place the nudge blocks).
3. **Generate a shareable report when done?** One answer covers every component in this batch. Default yes when the component's previous bundle has a report.

On yes: commit the plan (`plan: <NN-slug> v<N>` + the scorecard file), then execute.

## During execution

Unchanged doctrine — the plan resolves the big choices, but interpretive choices that arise mid-run (variable handling, exclusions, coding rules) still surface to the researcher before acting (CLAUDE.md rule 4), and decisions append to the log in real time. Multiple components run SEQUENTIALLY, full loop each, in this session — never parallel, never dispatched to subagents.

## The autopilot tail (after the work of one component ends)

Run these in order, without asking anything:

1. **Capture — agent-curated.** Follow `/research-plans:results <component>` steps 2, 4, 5 with the interview REPLACED by curation: artifact candidates come from the plan's Verification section, this session's outputs, and `discover`; you write titles, captions, and finding statements yourself, grouped under findings exactly as the interview would have. Stamp `"curatedBy": "agent"` into the manifest. Every other capture rule holds verbatim (provenance rule, producedBy honesty, journal-ready renders, late flags). The researcher's remedy for a mis-curated bundle is Reopen → recapture as r(N+1) — say so in the capture summary.
2. **Validate FIRST** (results.md step 6 verbatim — integrity + plan-conformance, mechanical status derivation). Bookkeeping waits until validation lands: a deviations-found result must interrupt BEFORE the tracker says done.
3. **Branch on the outcome matrix** (below): `deviations-found` → the deviation stop; anything else → continue.
4. **Finalize** (results.md step 7 mechanics), then **report** if the execute prompt pre-answered yes AND the bundle has substantive findings (the null-result gate holds) — proceed into `/research-plans:report <component> r<N>`; its marker records the validation state (schemaVersion 2).
5. **Bookkeeping (the tail carries these /sync jobs):** update the tracker row per the matrix + a one-line outcome note; append decision-log entries for session decisions that were never logged, each headed `## <YYYY-MM-DD HH:MM> (auto-captured)` — the board badges them; amending one later means APPENDING a corrective entry (the log stays append-only); check `results.py changed` for source drift; flag a component that grew past `references/split-criteria.md` as a proposed split (tracker rows, no dialogue).
   NOT carried (manual `/sync` only): hosted-comment pulls, adoption-cutoff handling, no-git evidence handling, full reconciliation of work done outside this loop.
6. **One commit suggestion** at the very end covering bundle + report + tracker + log (`plans: <NN-slug> executed — r<N> captured, validated` — do not run without approval). This is the ONLY commit prompt in the tail.
7. **Board, view-only:** open the board once on the bundle by proceeding into the full `/research-plans:board <NN-slug>:r<N>` workflow (routing stays live for comments/reopen; nothing waits on a verdict). With several components executed, open once at the end over everything.

## The deviation stop (the loop's only interruption)

Trigger: validation status `deviations-found`. Do NOT write the tracker status yet. Present the deviating steps/criteria with their evidence, then ask ONE AskUserQuestion with three remedies:

- **Revise the plan** — draft `v<N+1>` documenting the deviation (sync.md step 6 mechanics: Supersedes line, rp-model marker, no trailer), score the draft, finalize in the review room; after signing, RE-RUN validation against v<N+1> (it should now conform) and continue the tail from step 4.
- **Fix the work** — treat the deviations as defects: fix scripts, re-run, recapture as r<N+1> (trigger `redo-after-review`), re-validate, continue.
- **Accept and log** — append a decision-log entry (real timestamp, the deviation and why it stands); tracker status `done (validated)` does NOT apply — write `done (unvalidated)` with the note naming the accepted deviation; continue the tail from step 4.

## Outcome matrix (normative)

| Validation outcome | Tracker status | Tail behavior |
|---|---|---|
| `conforms` | `done (validated)` | continue |
| `conforms-with-amendments` | `done (validated)` — note names the amendment | continue |
| `deviations-found` | none until the deviation stop resolves | STOP (above) |
| `unverifiable` | `done (unvalidated)` — note says why | continue; surfaced on the board |
| `skipped` (opt-out / headless) | `done (unvalidated)` | continue |
| `not-applicable` (retrofit) | `done (retrofit)` | continue |
| integrity failed | (orthogonal — any row above) | advisory badge; never blocks |
| zero qualifying artifacts | no bundle | results.md rule: report honestly and stop |

## Loop closure

After the tail (all components), propose next steps from the tracker in one short message: the next `not started` row(s) in table order; when several are ready, offer batch planning (`/research-plans:plan <a> <b> …`); when every row is done/dropped, point at `/research-plans:renew`. Never start the next component without the researcher's word.

## Headless rules

No AskUserQuestion means: `/research-plans:execute` never auto-runs — it requires explicit arguments (`--model <id>`, `--report yes|no`, and an explicit go in the invocation); absent those, print what is needed and stop. The deviation stop cannot ask either: record the deviations, write NO tracker status, report, and stop — never auto-pick a remedy. Validation itself follows results.md's headless rule (`skipped` when the Task tool is unavailable).
```
